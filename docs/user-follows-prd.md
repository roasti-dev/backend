## Executive Summary

Roasti — социальная кофейная платформа. Сейчас нет механизма отслеживания авторов: пользователь не может подписаться на интересного автора и видеть его новые посты. Фича подписок создаёт социальный граф платформы и формирует персональную ленту контента.

Подписка односторонняя (как в Instagram): пользователь A подписывается на B, не требуя взаимности. Лента подписок возвращает посты авторов из подписок в хронологическом порядке. Публичный профиль отображает счётчики подписчиков и подписок, а также флаги `is_following` / `is_followed` для удобства клиента.

---

## Problem Statement

**Текущая ситуация**: Нет способа следить за конкретными авторами. Пользователь вынужден вручную заходить на профили интересных людей чтобы увидеть новый контент.

**Предлагаемое решение**: Система подписок с персональной лентой постов из подписок, счётчиками на профиле и уведомлением о новом подписчике.

**Бизнес-эффект**: Рост удержания — пользователи возвращаются чтобы увидеть новые посты авторов которых они читают. Формирование социального графа платформы.

---

## Success Metrics

**Ключевые метрики:**
- Рост числа возвращающихся сессий у подписанных пользователей
- Количество подписок на пользователя (глубина социального графа)

**Валидация**: Замер через сессионные логи и таблицу `follows` спустя 30 дней после запуска.

## User Personas

### Основной: Активный читатель
- **Роль**: Пользователь, который нашёл интересного автора через рекомендации или поиск
- **Цель**: Не пропускать новые посты любимых авторов
- **Боль**: Нет способа подписаться — нужно помнить имена авторов и заходить вручную

### Вторичный: Автор контента
- **Роль**: Пользователь, который регулярно публикует посты и рецепты
- **Цель**: Знать насколько его контент находит аудиторию
- **Боль**: Нет обратной связи о том сколько людей следит за его контентом

## User Stories & Acceptance Criteria

### Story 1: Подписаться / отписаться

**As a** авторизованный пользователь
**I want to** подписаться на другого пользователя или отписаться от него
**So that** я мог управлять своей лентой

**Acceptance Criteria:**
- [ ] `POST /api/v1/users/{user_id}/follow` — подписывает текущего пользователя на `user_id`
- [ ] `DELETE /api/v1/users/{user_id}/follow` — отписывает текущего пользователя от `user_id`
- [ ] Подписка идемпотентна: повторный `POST` возвращает 200, не ошибку
- [ ] Отписка идемпотентна: `DELETE` на незафоловленного пользователя возвращает 204
- [ ] Подписка на самого себя запрещена (400)
- [ ] Подписка на несуществующего пользователя возвращает 404
- [ ] Оба эндпоинта требуют авторизации (401 без токена)

### Story 2: Лента подписок

**As a** авторизованный пользователь
**I want to** видеть посты авторов на которых я подписан
**So that** я не пропускал новый контент

**Acceptance Criteria:**
- [ ] `GET /api/v1/posts?filter=following` возвращает посты авторов из подписок текущего пользователя
- [ ] Посты отсортированы по дате публикации (убывание, новые сверху)
- [ ] Если пользователь ни на кого не подписан или у подписок нет постов — пустой массив, статус 200
- [ ] Стандартная пагинация `page`/`limit`
- [ ] Эндпоинт требует авторизации (401 без токена)

### Story 3: Счётчики и статус на профиле

**As a** любой пользователь
**I want to** видеть счётчики подписчиков и подписок на профиле
**So that** я понимал популярность автора

**Acceptance Criteria:**
- [ ] `UserProfile` содержит поля `followers_count` и `following_count`
- [ ] `UserProfile` содержит `is_following` — подписан ли текущий пользователь на этот профиль
- [ ] `UserProfile` содержит `is_followed` — подписан ли владелец профиля на текущего пользователя
- [ ] Для анонимных запросов `is_following` и `is_followed` равны `false`
- [ ] Счётчики обновляются сразу при подписке/отписке

### Story 4: Списки подписчиков и подписок

**As a** авторизованный пользователь
**I want to** видеть список тех на кого я подписан и кто подписан на меня
**So that** я мог управлять своими подписками

**Acceptance Criteria:**
- [ ] `GET /api/v1/users/me/following` — список пользователей на которых подписан текущий пользователь
- [ ] `GET /api/v1/users/me/followers` — список подписчиков текущего пользователя
- [ ] Оба эндпоинта доступны только самому пользователю (401 без токена, 403 для чужого запроса)
- [ ] Стандартная пагинация `page`/`limit`
- [ ] Возвращают `UserPreview`

### Story 5: Уведомление о новом подписчике

**As a** пользователь
**I want to** получать уведомление когда кто-то подписывается на меня
**So that** я знал что моя аудитория растёт

**Acceptance Criteria:**
- [ ] При подписке создаётся уведомление для целевого пользователя
- [ ] Уведомление содержит информацию об авторе подписки (actor)
- [ ] Повторная подписка (идемпотентный запрос) не создаёт дублирующее уведомление

## Functional Requirements

### Архитектура хранения

Подписки хранятся с абстракцией через `target_type` для будущей расширяемости (например, подписка на организацию):

```sql
CREATE TABLE follows (
    id           TEXT PRIMARY KEY,
    follower_id  TEXT NOT NULL,
    target_id    TEXT NOT NULL,
    target_type  TEXT NOT NULL,
    created_at   DATETIME NOT NULL,
    UNIQUE (follower_id, target_id, target_type),
    FOREIGN KEY (follower_id) REFERENCES users(id)
);
```

### Эндпоинты

| Метод | Путь | Описание |
|-------|------|----------|
| `POST` | `/api/v1/users/{user_id}/follow` | Подписаться |
| `DELETE` | `/api/v1/users/{user_id}/follow` | Отписаться |
| `GET` | `/api/v1/posts?filter=following` | Лента подписок |
| `GET` | `/api/v1/users/me/following` | Кого читаю |
| `GET` | `/api/v1/users/me/followers` | Кто читает меня |

### Out of Scope
- Подписки на рецепты или зерна
- Рекомендации "кого подписаться"
- Приватные аккаунты с запросом на подписку
- Взаимные подписки (mutual follow) как отдельная сущность
- Уведомления о новых постах авторов из подписок (push)

## Technical Constraints

### Performance
- Запросы к БД укладываются в общий бюджет ответа API
- Индекс на `follows(follower_id, target_type)` для быстрой выборки подписок

### Security
- Все операции с подписками требуют авторизации
- Списки followers/following доступны только самому пользователю

### Integration
- **Таблица `users`**: добавляются вычисляемые поля `followers_count`, `following_count` в `UserProfile`
- **Таблица `posts`**: `filter=following` джойнит `follows` для фильтрации
- **Система уведомлений**: новое событие `follow` публикует уведомление целевому пользователю

## MVP Scope & Phasing

### Phase 1: MVP
- Подписка / отписка с идемпотентностью
- Лента постов из подписок (`filter=following`)
- Счётчики + `is_following` / `is_followed` на профиле
- Списки followers / following (только для себя)
- Уведомление о новом подписчике

### Phase 2: Улучшения
- Подписки на организации (уже поддержано схемой через `target_type`)
- Уведомления о новых постах авторов из подписок

### Будущее
- Рекомендации "кого подписаться" на основе графа
- Приватные аккаунты

## Risk Assessment

| Риск | Вероятность | Влияние | Митигация |
|------|------------|---------|-----------|
| Лента пустая на старте — мало контента | Высокая | Среднее | Комбинировать с рекомендациями рецептов/пользователей |
| Дублирующиеся уведомления при идемпотентных подписках | Средняя | Низкое | Проверять существование подписки перед созданием уведомления |
| Тяжёлый JOIN при большом числе подписок | Низкая | Среднее | Индекс на `follows(follower_id, target_type)` |
