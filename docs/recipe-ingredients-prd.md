## Executive Summary

Сейчас пользователи Roasti вынуждены перечислять ингредиенты внутри текстового описания рецепта — это неструктурировано и неудобно для восприятия. Особенно остро проблема ощущается для сложных напитков, где важно сразу понять состав и пропорции.

Фича добавляет к рецептам опциональный структурированный список ингредиентов (название, количество, единица измерения). Список хранится и возвращается как часть рецепта, редактируется вместе с ним. Порядок ингредиентов задаётся клиентом; пропорциональное масштабирование порций также реализуется на клиенте — бэкенд хранит базовые значения.

---

## Problem Statement

**Текущая ситуация**: Ингредиенты пишутся произвольным текстом в описании рецепта. Нет единого формата — сложно быстро оценить состав напитка.

**Предлагаемое решение**: Структурированный массив ингредиентов (название + количество + единица) как часть модели рецепта.

**Бизнес-эффект**: Рецепты становятся удобнее для чтения и воспроизведения. Основная ценность — для сложных напитков с несколькими компонентами.

## Success Metrics

Формальные метрики пока не отслеживаются. Качественный индикатор успеха: пользователи перестают перечислять ингредиенты в свободном тексте описания и переходят на структурированный список.

## User Personas

### Primary: Кофе-энтузиаст / бариста
- **Роль**: Создаёт и публикует рецепты напитков
- **Цель**: Поделиться точным составом и пропорциями напитка
- **Боль**: Приходится писать ингредиенты в описании — неаккуратно выглядит, неудобно читать

### Secondary: Читатель рецепта
- **Роль**: Изучает рецепт перед приготовлением
- **Цель**: Быстро понять, что нужно для напитка
- **Боль**: Ингредиенты «спрятаны» в тексте описания

## User Stories & Acceptance Criteria

### Story 1: Добавление ингредиентов к рецепту

**As a** пользователь, создающий рецепт  
**I want to** добавить список ингредиентов с названием, количеством и единицей  
**So that** читатели сразу видят полный состав напитка

**Acceptance Criteria:**
- [ ] Список ингредиентов опционален — рецепт без ингредиентов создаётся без ошибок
- [ ] Каждый ингредиент содержит: название (обязательно), количество (опционально), единицу измерения (опционально, свободный текст)
- [ ] Название ингредиента не может быть пустой строкой
- [ ] Количество, если указано, должно быть положительным числом
- [ ] Максимум 50 ингредиентов в одном рецепте
- [ ] Порядок ингредиентов сохраняется в том виде, в котором прислан клиентом

### Story 2: Редактирование ингредиентов

**As a** автор рецепта  
**I want to** изменить список ингредиентов при редактировании рецепта  
**So that** информация о составе остаётся актуальной

**Acceptance Criteria:**
- [ ] `PUT /recipes/{id}` принимает полный новый массив ингредиентов и полностью заменяет старый
- [ ] Передача пустого массива `[]` удаляет все ингредиенты рецепта

### Story 3: Просмотр ингредиентов

**As a** читатель рецепта  
**I want to** видеть список ингредиентов на странице рецепта  
**So that** я могу быстро оценить состав и подготовить всё необходимое

**Acceptance Criteria:**
- [ ] Все эндпоинты, возвращающие рецепт, включают массив `ingredients`
- [ ] Если ингредиентов нет — возвращается пустой массив `[]`

## Functional Requirements

### Core Features

**Структура ингредиента**
- `name` (string, обязательно) — название ингредиента, свободный текст
- `amount` (number, опционально) — количество; если не указано — «по вкусу»
- `unit` (string, опционально) — единица измерения, свободный текст (мл, г, oz и т.д.)
- Порядок в массиве = порядок отображения

**Редактирование**
- Ингредиенты — часть `RecipePayload`, используемого в `POST /recipes` и `PUT /recipes/{id}`
- Полное замещение массива при каждом обновлении рецепта
- Нет отдельных эндпоинтов для управления отдельными ингредиентами

**Отображение**
- Ингредиенты возвращаются во всех эндпоинтах, возвращающих рецепт (`GET /recipes/{id}`, листинги и т.д.)
- Единая модель рецепта — без разделения на «краткую» и «полную» версию

### Out of Scope
- Поиск и фильтрация рецептов по ингредиентам
- Расчёт пищевой ценности
- Предустановленный справочник единиц измерения
- Масштабирование порций на бэкенде (реализуется на клиенте)
- Ингредиенты в статьях (рецепты с ингредиентами могут прикрепляться к статьим — этого достаточно)

## Technical Constraints

### Validation
- `name`: непустая строка после trim
- `amount`: положительное число (> 0), если указано
- Максимум 50 ингредиентов на рецепт

### Performance
- Ингредиенты хранятся в отдельной таблице, связанной с рецептом по `recipe_id`
- Загружаются для всех эндпоинтов, возвращающих рецепт (единая модель)

### Security
- Редактировать ингредиенты может только автор рецепта (наследует ограничения `PATCH /recipes/{id}`)

## MVP Scope & Phasing

### Phase 1: MVP
- Схема БД: таблица `recipe_ingredients` (id, recipe_id, name, amount, unit, position)
- API: поле `ingredients` в `GET /recipes/{id}` и `PATCH /recipes/{id}`
- Валидация: пустое название, отрицательное количество, лимит 50

**MVP Definition**: Пользователь может добавить, просмотреть и обновить список ингредиентов рецепта.

### Future Considerations
- Фильтрация рецептов по наличию конкретных ингредиентов
- Стандартизированный справочник единиц (для поддержки масштабирования на бэкенде)
