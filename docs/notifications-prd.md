## Executive Summary

Roasti пользователи, создающие рецепты и посты, не получают никакой обратной связи о том, как другие реагируют на их контент. Отсутствие социальной петли обратной связи снижает мотивацию авторов публиковать новый контент и возвращаться в приложение.

Фича добавляет in-app систему уведомлений: бейджик с количеством непрочитанных и экран с историей уведомлений. На первом этапе охватываются лайки и комментарии к рецептам и постам. Push-уведомления для мобильных клиентов запланированы на Phase 2.

---

## Problem Statement

**Текущая ситуация**: Авторы рецептов и постов не знают, когда кто-то лайкает или комментирует их контент. Единственный способ увидеть реакцию — вручную зайти на страницу своего рецепта/поста.

**Предлагаемое решение**: In-app центр уведомлений с историей активности, автоматической маркировкой как прочитанных при открытии экрана и лёгким endpoint для polling'а бейджика.

**Ожидаемый эффект**: Авторы осведомлены о реакции на контент и возвращаются в приложение, когда видят непрочитанные уведомления.

---

## Success Metrics

Метрики запланированы к определению после запуска MVP. На этапе разработки успех определяется через корректную функциональность:

- Уведомления создаются для всех поддерживаемых событий без дублей
- Бейджик не показывает прочитанные уведомления
- Нет self-уведомлений (лайк/комментарий на собственный контент)

Количественные KPI (retention, engagement) — в рамках отдельного post-launch анализа.

## User Personas

### Primary: Автор контента

- **Роль**: Пользователь, публикующий рецепты и/или посты
- **Цели**: Знать, что его контент замечают; получать мотивацию публиковать больше
- **Pain Points**: Пропускает реакции на контент, не зная что они есть
- **Клиенты**: Мобильное приложение (KMP) и веб (SvelteKit)

### Secondary: Активный читатель

- **Роль**: Пользователь, который лайкает и комментирует чужой контент
- **Цели**: Взаимодействовать с контентом
- **Отношение к фиче**: Триггерит уведомления для авторов, сам их не получает в рамках MVP

## User Stories & Acceptance Criteria

### Story 1: Уведомление о лайке

**As a** автор рецепта или поста  
**I want to** получать уведомление, когда кто-то лайкает мой контент  
**So that** я знаю, что моя работа ценится

**Acceptance Criteria:**
- [ ] Уведомление создаётся, когда пользователь A лайкает рецепт/пост пользователя B (A ≠ B)
- [ ] Уведомление содержит: тип (`like_recipe` / `like_post`), id актора, id сущности
- [ ] При анлайке уведомление не создаётся и существующее не удаляется
- [ ] Лайк собственного контента не порождает уведомление

### Story 2: Уведомление о комментарии

**As a** автор рецепта или поста  
**I want to** получать уведомление, когда кто-то оставляет комментарий  
**So that** я могу вступить в диалог

**Acceptance Criteria:**
- [ ] Уведомление создаётся, когда пользователь A комментирует рецепт/пост пользователя B (A ≠ B)
- [ ] Только автор контента получает уведомление (не другие участники треда)
- [ ] Комментарий к собственному контенту не порождает уведомление

### Story 3: Список уведомлений

**As a** пользователь  
**I want to** видеть историю уведомлений  
**So that** я могу просмотреть пропущенную активность

**Acceptance Criteria:**
- [ ] Список отсортирован от нового к старому
- [ ] Каждый элемент содержит: тип, данные актора (id, username, avatar_id), entity_id, read_at, created_at
- [ ] Клиент формирует текст уведомления самостоятельно на основе типа и данных актора
- [ ] Поддерживается пагинация (limit/offset)

### Story 4: Бейджик непрочитанных

**As a** пользователь  
**I want to** видеть количество непрочитанных уведомлений  
**So that** я знаю, когда заходить в центр уведомлений

**Acceptance Criteria:**
- [ ] Доступен лёгкий endpoint, возвращающий только число непрочитанных
- [ ] Клиент может polling'ить этот endpoint в фоне без лишней нагрузки

### Story 5: Маркировка как прочитанных

**As a** пользователь  
**I want to** чтобы уведомления отмечались прочитанными при открытии экрана  
**So that** бейджик пропадал и я мог видеть, что уже видел

**Acceptance Criteria:**
- [ ] Все непрочитанные отмечаются прочитанными при вызове `POST /notifications/read_all`
- [ ] Клиент вызывает этот endpoint при открытии экрана уведомлений
- [ ] После вызова `unread_count` возвращает 0

## Functional Requirements

### Поддерживаемые типы уведомлений (MVP)

| Тип | Триггер |
|---|---|
| `like_recipe` | Лайк на рецепт |
| `like_post` | Лайк на пост |
| `comment_recipe` | Комментарий к рецепту |
| `comment_post` | Комментарий к посту |

### API

**`GET /api/v1/notifications`**  
Возвращает пагинированный список уведомлений текущего пользователя.  
Параметры: `page`, `limit`. Ответ содержит список уведомлений и мета-данные пагинации.

**`GET /api/v1/notifications/unread_count`**  
Возвращает количество непрочитанных уведомлений. Лёгкий endpoint для фонового polling'а.

**`POST /api/v1/notifications/read_all`**  
Отмечает все непрочитанные уведомления текущего пользователя как прочитанные. Возвращает 204.

### Модель уведомления

```
id          — уникальный идентификатор
type        — тип (like_recipe | like_post | comment_recipe | comment_post)
actor       — UserPreview (id, username, avatar_id)
entity_id   — id рецепта или поста
read_at     — дата прочтения (null если непрочитано)
created_at  — дата создания
```

### Правила создания

- Уведомление создаётся асинхронно через event bus (не блокирует основной запрос)
- Актор == владелец контента → уведомление не создаётся
- Анлайк → уведомление не создаётся, существующее уведомление о лайке не удаляется
- Лимит хранения не установлен (MVP)

### Polling-стратегия (клиент)

- **Фоновый polling**: `GET /unread_count` раз в 30–60 секунд → обновить бейджик
- **При открытии экрана**: `GET /notifications` → отобразить список, затем `POST /read_all`

### Out of Scope (MVP)

- Push-уведомления (мобильные / браузерные)
- Уведомления о подписчиках (фича подписчиков не реализована)
- Уведомления на ответы в треде (только автор контента)
- Настройки уведомлений (включить/выключить по типу)
- Удаление отдельных уведомлений
- Лимит/TTL хранения уведомлений

## Technical Constraints

### Performance
- `GET /unread_count` — максимально лёгкий запрос (один `COUNT` по индексу)
- Создание уведомления асинхронно через event bus — не влияет на latency основных операций

### Security
- Все три endpoint требуют аутентификации
- Пользователь видит только свои уведомления (фильтрация по `user_id`)

### Integration

- **Event Bus**: Уведомления создаются как подписчик на события `RecipeLikeToggled`, `PostLikeToggled`, `RecipeCommentCreated`, `PostCommentCreated`
- **SQLite**: Новая таблица `notifications` с FK на `users`
- **Firebase Auth**: Аутентификация стандартная, как у всех остальных endpoint'ов

### Technology Stack
- Backend: Go + SQLite + squirrel query builder
- Клиенты: KMP (mobile), SvelteKit (web)
- Генерация кода: OpenAPI-first, `make oapi` после изменений spec

## MVP Scope & Phasing

### Phase 1: MVP

- Таблица `notifications` в БД
- Event handlers для 4 типов событий
- `GET /notifications`, `GET /notifications/unread_count`, `POST /notifications/read_all`
- Интеграция с существующим event bus

### Phase 2: Push-уведомления (мобильные)

- Поддержка FCM / APNs токенов для KMP-клиентов
- Отправка push при создании уведомления (параллельно с in-app)
- Таблица устройств / токенов

### Future Considerations

- Уведомления о новых подписчиках (после реализации follows)
- Уведомления на ответы в треде (reply to comment)
- Настройки уведомлений по типу
- Retention policy (TTL или лимит количества на пользователя)
- Аналитика и KPI (engagement, retention)

## Risk Assessment

| Риск | Вероятность | Влияние | Митигация |
|---|---|---|---|
| Дублирование уведомлений при повторной отправке события | Низкая | Средняя | Идемпотентность на уровне event handler: проверять `Liked == true` перед созданием |
| Рост таблицы при большой активности | Низкая (ранний этап) | Средняя | Добавить retention policy в Phase 2 |
| Push-архитектура несовместима с текущей моделью | Низкая | Высокая | Таблица уведомлений спроектирована с учётом расширения |

## Appendix

### Glossary

- **Actor** — пользователь, совершивший действие (лайкнул, прокомментировал)
- **Entity** — рецепт или пост, к которому относится уведомление
- **Polling** — периодический опрос API клиентом вместо постоянного соединения
- **Event Bus** — внутренний асинхронный канал событий домена (`internal/events`)
