## Executive Summary

Посты — это социальная лента в Roasti, позволяющая пользователям общаться вокруг темы кофе: обсуждать зёрна и обжарщиков, делиться впечатлениями и фотографиями, задавать вопросы, публиковать гайды и обзоры. В отличие от рецептов (структурированный контент), посты — свободный формат, как в Reddit: блоки текста, изображений и ссылок на рецепты в произвольном порядке.

Фича создаёт пространство для трёх типов авторов: обычных пользователей, обжарщиков и инфлюенсеров. Это увеличивает время в приложении и формирует сообщество вокруг платформы.

---

## Problem Statement

**Текущая ситуация:** Roasti предоставляет инструменты для хранения и просмотра рецептов, но не даёт пользователям возможности общаться, делиться нерецептурным контентом или задавать вопросы сообществу.

**Предлагаемое решение:** Глобальная лента постов со свободным форматом контента, лайками и комментариями.

**Бизнес-эффект:** Рост вовлечённости и времени в приложении; платформа становится точкой притяжения для кофейного сообщества, а не только инструментом для рецептов.

---

## Success Metrics

Конкретные числовые цели не определены на этом этапе. Рекомендуется начать с отслеживания:

- Количество созданных постов за период (активность авторов)
- Количество комментариев на пост (глубина вовлечённости)
- Доля постов с хотя бы одним лайком (резонанс контента)
- Доля активных пользователей, открывших ленту (охват фичи)

**Валидация:** Провести ревью метрик через 4 недели после запуска.

## User Personas

### Обычный пользователь
- **Цели:** Поделиться впечатлениями от новых зёрен, спросить совет, посмотреть что пьют другие
- **Поведение:** Читает ленту, изредка создаёт посты, комментирует

### Обжарщик
- **Цели:** Презентовать новые лоты, публиковать гайды по своим зёрнам
- **Поведение:** Регулярно создаёт посты с текстом, фото и ссылками на рецепты

### Инфлюенсер / Эксперт
- **Цели:** Публиковать обзоры, коллаборации, образовательный контент
- **Поведение:** Создаёт структурированные длинные посты с несколькими блоками

## User Stories & Acceptance Criteria

### Story 1: Создание поста

**As a** авторизованный пользователь  
**I want to** создать пост с заголовком и набором блоков (текст, фото, ссылка на рецепт)  
**So that** поделиться контентом с сообществом

**Acceptance Criteria:**
- [ ] Пост требует заголовок; список блоков может быть пустым
- [ ] Поддерживаются три типа блоков: текст, изображения (одно и больше), ссылка на рецепт
- [ ] Блоки располагаются в заданном порядке
- [ ] Пост сразу публичный — черновиков нет (v1)
- [ ] Любой авторизованный пользователь может создать пост без дополнительной верификации

### Story 2: Просмотр ленты

**As a** пользователь  
**I want to** листать глобальную ленту постов  
**So that** быть в курсе активности сообщества

**Acceptance Criteria:**
- [ ] Лента отдаёт посты всех пользователей, отсортированных по дате (новые первые)
- [ ] Лента пагинирована
- [ ] Каждый пост в ленте содержит: заголовок, блоки, автора, дату, количество лайков, `is_liked`, список комментариев (см. `comments-prd.md`)

### Story 3: Редактирование и удаление поста

**As a** автор поста  
**I want to** отредактировать или удалить свой пост  
**So that** поддерживать актуальность своего контента

**Acceptance Criteria:**
- [ ] Только автор может редактировать и удалять пост
- [ ] Редактирование доступно без ограничений по времени
- [ ] Редактирование возвращает обновлённый объект поста

### Story 4: Лайк поста

**As a** пользователь (в том числе автор поста)  
**I want to** лайкнуть или убрать лайк с поста  
**So that** выразить отношение к контенту

**Acceptance Criteria:**
- [ ] Toggle-логика: повторный запрос убирает лайк
- [ ] Ответ содержит `liked` и актуальный `likes_count`
- [ ] Лайкнуть может любой авторизованный пользователь, включая автора

### Story 5: Лента лайкнутых постов

**As a** пользователь  
**I want to** просмотреть посты, которые я лайкнул  
**So that** вернуться к понравившемуся контенту

**Acceptance Criteria:**
- [ ] `GET /api/v1/users/{user_id}/likes?type=post` возвращает пагинированный список `LikedPost`
- [ ] Каждый элемент содержит `liked_at` и полный объект поста

## Functional Requirements

### Блоки поста

Пост состоит из упорядоченного списка блоков. Каждый блок — объект с полем `type` и соответствующими данными:

| Тип | Данные | Описание |
|-----|--------|----------|
| `text` | `text: string` | Текстовый параграф |
| `images` | `images: string[]` | Массив ID изображений (одно и более) |
| `recipe` | `recipe_id: string` | Ссылка на рецепт в Roasti |

Неактивные поля блока отсутствуют в ответе (не передаются как `null`).

**Обработка недоступного рецепта:**  
Бэкенд резолвит все `recipe_id` в блоках при отдаче поста и добавляет поле `recipe_status: "available" | "unavailable"`. Клиент показывает плейсхолдер если статус `unavailable` — без дополнительных запросов. Причина недоступности (удалён или стал приватным) не раскрывается.

**Лимиты блоков и изображений:** не установлены в v1. Необходимо вернуться к этому вопросу перед масштабированием.

### Видимость постов

Все посты публичные. Авторизация не влияет на видимость.

### Права доступа

| Действие | Кто может |
|----------|-----------|
| Создать пост | Любой авторизованный пользователь |
| Редактировать пост | Только автор |
| Удалить пост | Только автор |
| Лайкнуть пост | Любой авторизованный пользователь |
### Модерация (v1)

Минимальный уровень модерации, обязательный до публичного запуска:

- **Репорт** — пользователь может пожаловаться на пост или комментарий. Репорт отправляется на почту команды.
- **Админское удаление** — внутренний эндпоинт для удаления любого поста или комментария без UI. Доступен только команде.

Более сложная модерация (очередь репортов, автофильтры, баны) — в будущих итерациях.

## API Contract

| Метод | Путь | Статус | Описание |
|-------|------|--------|----------|
| `GET` | `/api/v1/posts` | 200 | Лента постов (пагинация) |
| `POST` | `/api/v1/posts` | 201 | Создать пост |
| `GET` | `/api/v1/posts/{post_id}` | 200 | Получить пост |
| `PUT` | `/api/v1/posts/{post_id}` | 200 | Обновить пост |
| `DELETE` | `/api/v1/posts/{post_id}` | 204 | Удалить пост |
| `POST` | `/api/v1/posts/{post_id}/like` | 200 | Toggle лайк |
| `POST` | `/api/v1/posts/{post_id}/comments` | 201 | Добавить комментарий (см. `comments-prd.md`) |
| `GET` | `/api/v1/posts/{post_id}/comments` | 200 | Комментарии поста (см. `comments-prd.md`) |
| `GET` | `/api/v1/users/{user_id}/likes?type=post` | 200 | Лайкнутые посты пользователя |

## Out of Scope (v1)

- **Черновики постов** — посты сразу публичные; черновики рассмотреть в следующей итерации
- **Приватные посты** — видимость и гранулярный контроль доступа в будущем
- **Поиск** — полнотекстовый поиск по ленте (поля TBD) в следующей итерации
- **Персонализированная лента / подписки** — следующая итерация
- **Комментарии** — правила, права и поведение описаны в `comments-prd.md`
- **Верификация / роли** — все пользователи равны в v1
- **Лимиты блоков и изображений** — установить лимиты до масштабирования

## Risk Assessment

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Посты со ссылками на удалённые/приватные рецепты | Высокая | Среднее | Плейсхолдер на клиенте; механизм обнаружения недоступности в API |
| Спам и злоупотребления | Средняя | Высокое | Репорты + админское удаление реализованы в v1 до публичного запуска |
| Отсутствие лимитов на блоки | Средняя | Среднее | Установить лимиты до масштабирования |
| Производительность при встроенных комментариях | Низкая | Среднее | Мониторить; пагинация и оптимизация описаны в `comments-prd.md` |

## Dependencies

- Система загрузки изображений (uploads) — для блоков типа `images`
- API рецептов — для блоков типа `recipe` и отображения превью
- Firebase Auth — авторизация всех операций с постами
