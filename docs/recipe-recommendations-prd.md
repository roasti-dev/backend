## Executive Summary

Roasti — социальная кофейная платформа, где пользователи делятся рецептами заваривания. Сейчас нет механизма обнаружения рецептов кроме ручного поиска или перехода с профиля автора. Фича рекомендаций рецептов позволит активным пользователям находить популярный и свежий контент без лишних усилий.

Ранжирование строится на двух сигналах: лайки за последние 30 дней (популярность) и бонус за свежесть для рецептов моложе 30 дней (даёт новым рецептам шанс попасть в выдачу). Пользователь видит только то, что ещё не лайкал и что создали другие авторы — это делает ленту рекомендаций полностью новой для него.

---

## Problem Statement

**Текущая ситуация**: Нет механизма обнаружения рецептов. Пользователь видит только рецепты авторов, на профили которых он перешёл вручную.

**Предлагаемое решение**: Параметр `sort=recommended` на эндпоинте `GET /api/v1/recipes` возвращает публичные рецепты, отсортированные по комбинированному скору из лайков за 30 дней и бонуса за свежесть, исключая собственные рецепты пользователя и уже лайкнутые им.

**Бизнес-эффект**: Рост вовлечённости активных пользователей — они находят новый интересный контент и возвращаются на платформу.

---

## Success Metrics

**Ключевые метрики:**
- Рост числа лайков на рецептах из рекомендаций
- Рост переходов на профили авторов из рекомендованных рецептов

**Валидация**: Замер через SQL-запросы к таблицам `likes` и сессионным логам спустя 30 дней после запуска.

## User Personas

### Основной: Активный читатель
- **Роль**: Пользователь, который регулярно лайкает и читает чужие рецепты
- **Цель**: Находить новые интересные рецепты без ручного поиска
- **Боль**: Видит одних и тех же авторов, не знает о новых рецептах на платформе

## User Stories & Acceptance Criteria

### Story 1: Лента рекомендованных рецептов

**As a** авторизованный пользователь
**I want to** получить список рекомендованных рецептов
**So that** я мог найти новые рецепты которые ещё не видел

**Acceptance Criteria:**
- [ ] `GET /api/v1/recipes?sort=recommended` возвращает список `Recipe` с пагинацией
- [ ] Список отсортирован по убыванию скора: `лайки за 30 дней + бонус за свежесть`
- [ ] Рецепты текущего пользователя отсутствуют в результатах
- [ ] Рецепты, которые пользователь уже лайкнул, отсутствуют в результатах
- [ ] Возвращаются только публичные рецепты
- [ ] Эндпоинт требует авторизации (401 без токена)
- [ ] Пагинация через стандартные `page`/`limit` параметры

### Story 2: Пустой результат

**As a** авторизованный пользователь
**I want to** получить корректный ответ когда рекомендаций нет
**So that** приложение не сломалось

**Acceptance Criteria:**
- [ ] Если нет подходящих рецептов — возвращается пустой массив с `total: 0`
- [ ] HTTP статус 200, не ошибка

## Functional Requirements

### Алгоритм ранжирования

**Формула скора**:
```
score = likes_30d + freshness_bonus
```

- `likes_30d` — количество лайков на рецепте с `created_at >= NOW() - 30 days`
- `freshness_bonus` — фиксированная константа (по умолчанию **5**), применяется если рецепт создан менее 30 дней назад; иначе 0

Значение константы подбирается эмпирически по результатам наблюдений.

**Фильтрация**:
- Только публичные рецепты (`public = true`)
- Исключить рецепты текущего пользователя (`author_id != current_user_id`)
- Исключить рецепты, которые текущий пользователь уже лайкнул

**Поля ответа**: полный объект `Recipe`

### Out of Scope
- Анонимный доступ к рекомендациям
- Персонализация по методу заваривания на основе истории пользователя
- Весовые коэффициенты давности лайков внутри 30-дневного окна
- Кэширование результатов

## Technical Constraints

### Performance
- Запрос к БД должен укладываться в общий бюджет ответа API

### Security
- Эндпоинт доступен только авторизованным пользователям

### Integration
- **Существующий `GET /api/v1/recipes`**: добавляется новое значение параметра `sort`
- **Таблица `likes`**: фильтр по `target_type = 'recipe'` и `created_at >= NOW() - 30 days`
- **Таблица `recipes`**: фильтр по `public = true` и `author_id`

---

## MVP Scope & Phasing

### Phase 1: MVP
- Параметр `sort=recommended` на `GET /api/v1/recipes`
- Ранжирование: лайки за 30 дней + фиксированный бонус за свежесть
- Исключение собственных и уже лайкнутых рецептов
- Стандартная пагинация

### Phase 2: Улучшения (после запуска)
- Анализ константы freshness_bonus — подбор оптимального значения
- Персонализация по методу заваривания на основе истории лайков пользователя

### Будущее
- Рекомендации на основе пересечения вкусовых предпочтений
- Относительный бонус за свежесть (убывает по мере старения рецепта)

## Risk Assessment

| Риск | Вероятность | Влияние | Митигация |
|------|------------|---------|-----------|
| Мало данных на старте — рекомендации пустые или повторяются | Высокая | Среднее | Seed-данные, постепенный рост аудитории |
| Константа freshness_bonus подобрана неудачно | Средняя | Среднее | Начать с 5, наблюдать и корректировать |
| Тяжёлый SQL при большом числе лайков | Низкая | Среднее | Индекс на `likes(target_type, created_at)` |
