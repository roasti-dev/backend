## Executive Summary

Roasti — социальная кофейная платформа, где пользователи делятся рецептами, постами и зёрнами. Сейчас нет способа открыть новых авторов кроме случайного поиска. Фича рекомендаций позволит пользователям находить популярных авторов рецептов и увеличит глубину взаимодействия с платформой.

Рекомендации строятся на основе суммарного количества лайков на публичных рецептах автора за последние 30 дней. Это качественный сигнал: автор с двумя рецептами и 50 лайками релевантнее автора с 20 рецептами и 5 лайками. Свежее окно в 30 дней позволяет новым активным авторам попадать в рекомендации наравне с ветеранами.

---

## Problem Statement

**Текущая ситуация**: Нет механизма обнаружения авторов. Пользователь может найти другого только перейдя с рецепта или поста.

**Предлагаемое решение**: Параметр `sort=recommended` на эндпоинте `GET /api/v1/users` возвращает авторов, отсортированных по суммарному количеству лайков на их публичных рецептах за последние 30 дней.

**Бизнес-эффект**: Рост вовлечённости — пользователи находят интересных авторов и возвращаются за новым контентом.

---

## Success Metrics

**Ключевые метрики:**
- Рост числа переходов на профили пользователей
- Увеличение лайков на рецептах авторов из рекомендаций

**Валидация**: Замер через SQL-запросы к таблицам `likes` и сессионным логам спустя 30 дней после запуска.

**So that** я мог найти интересных людей для чтения

## Functional Requirements

### Алгоритм ранжирования

**Запрос**: Для каждого пользователя считается сумма лайков на его публичных рецептах, созданных или полученных в течение последних 30 дней. Пользователи сортируются по убыванию этой суммы.

**Фильтрация**:
- Исключить текущего пользователя
- Исключить пользователей без публичных рецептов
- Учитывать только лайки с `created_at >= NOW() - 30 days`

**Поля ответа**: `UserPreview` — id, username, name, avatar

### Out of Scope
- Персонализация на основе предпочтений текущего пользователя
- Рекомендации на основе постов или зёрен
- Кэширование результатов
- Весовые коэффициенты давности лайков внутри 30-дневного окна

## Technical Constraints

### Performance
- Запрос к БД должен укладываться в общий бюджет ответа API

### Security
- Эндпоинт доступен только авторизованным пользователям

### Integration
- **Существующий `GET /api/v1/users`**: добавляется новое значение параметра `sort`
- **Таблица `likes`**: используется поле `created_at` для фильтрации по периоду
- **Таблица `recipes`**: фильтр по `public = true` и `author_id`

## MVP Scope & Phasing

### Phase 1: MVP
- Параметр `sort=recommended` на `GET /api/v1/users`
- Ранжирование по лайкам на публичных рецептах за 30 дней
- Исключение текущего пользователя и авторов без рецептов
- Стандартная пагинация

### Phase 2: Улучшения (после запуска)
- Анализ данных: смотрим на что реагируют пользователи
- Возможное расширение сигнала — лайки постов, зёрен

### Будущее
- Персонализированные рекомендации на основе собственных лайков пользователя
- Рекомендации авторов по пересечению интересов
