## Executive Summary

Roasti нужна чёткая, уважительная к пользователям модель удаления и редактирования комментариев. Основной принцип: решение пользователя удалить свой комментарий — его право, но оно не должно незаметно уничтожать контекст для тех, кто ответил. Удаление постоянно и необратимо, однако нода комментария остаётся видимой как плейсхолдер — сохраняя целостность дерева ответов.

---

## Problem Statement

**Текущая ситуация:** В таблице комментариев нет механизма удаления. Физическое удаление каскадно и незаметно уничтожает все ответы под комментарием — разрывая обсуждение.

**Предлагаемое решение:** Мягкое удаление (soft delete) через timestamp `deleted_at`. Нода комментария остаётся в дереве; текст и личность автора скрыты от всех клиентов. Правки отслеживаются через `updated_at`.

**Бизнес-эффект:** Пользователи управляют своим контентом без риска непреднамеренных побочных эффектов. Обсуждения остаются связными даже после удалений.

## User Personas

### Основной: кофейный энтузиаст (автор комментария)
- **Цели:** Делиться мнением, участвовать в обсуждении, исправлять ошибки в своих комментариях
- **Боли:** Написал что-то по ошибке; хочет удалить, не сломав чужой ответ под своим комментарием

### Вторичный: администратор Roasti
- **Цели:** Удалять неприемлемый или нарушающий правила контент
- **Технический уровень:** Член команды с доступом к внутреннему API

## User Stories & Acceptance Criteria

### Story 1: Автор удаляет свой комментарий

**As a** автор комментария  
**I want to** удалить свой комментарий  
**So that** он больше не был виден другим

**Acceptance Criteria:**
- [ ] Только автор комментария может удалить его (HTTP 403 для остальных)
- [ ] После удаления нода комментария остаётся в дереве с установленным `deleted_at`
- [ ] API возвращает плейсхолдер вместо реального содержимого (текст и автор скрыты)
- [ ] Все ответы под удалённым комментарием остаются видимыми и незатронутыми
- [ ] Удаление необратимо — эндпоинта восстановления нет

### Story 2: Администратор удаляет комментарий

**As an** администратор  
**I want to** удалить комментарий, нарушающий правила  
**So that** он больше не был виден пользователям

**Acceptance Criteria:**
- [ ] Администратор может удалить любой комментарий вне зависимости от авторства
- [ ] Результирующее поведение идентично удалению автором (тот же плейсхолдер `[deleted]`)
- [ ] Дерево ответов сохраняется

### Story 3: Автор редактирует свой комментарий

**As a** автор комментария  
**I want to** отредактировать свой комментарий после публикации  
**So that** исправить ошибки или уточнить мысль

**Acceptance Criteria:**
- [ ] Только автор комментария может редактировать его (HTTP 403 для остальных)
- [ ] Временного ограничения на редактирование нет
- [ ] `updated_at` обновляется при каждом редактировании
- [ ] Ответы API включают `updated_at`, чтобы клиент мог показать индикатор «изменено»
- [ ] Удалённый комментарий нельзя редактировать

### Story 4: Аккаунт пользователя удалён

**As the** система  
**When** аккаунт пользователя удалён  
**I need to** мягко удалить все его комментарии

**Acceptance Criteria:**
- [ ] Удаление аккаунта инициирует soft delete всех комментариев этого пользователя
- [ ] Мягко удалённые комментарии отображаются по той же логике плейсхолдера `[deleted]`
- [ ] Деревья ответов под этими комментариями сохраняются

## Functional Requirements

### Поведение при удалении

| Сценарий | Поведение |
|----------|-----------|
| Комментарий имеет ответы | Нода остаётся, показывается плейсхолдер |
| Комментарий не имеет ответов | Нода остаётся, показывается плейсхолдер |
| Удаляет автор | Устанавливается `deleted_at`, данные сохраняются в БД |
| Удаляет администратор | То же самое |
| Аккаунт удалён | Все комментарии пользователя мягко удаляются |

**Содержимое плейсхолдера** (возвращается API, когда `deleted_at IS NOT NULL`):
- `text` — скрыт (не возвращается)
- `author` — скрыт (не возвращается)
- `created_at` — возвращается (для позиционирования в хронологии)
- `id`, `parent_id`, `target_id` — возвращаются (для структуры дерева)

### Поведение при редактировании

- Доступно автору в любое время
- Недоступно для удалённых комментариев
- `updated_at` отражает временну́ю метку последнего редактирования

### Out of Scope

- Восстановление / отмена удаления комментария
- Очередь модерации и репорты (запланированы на следующие итерации)
- Разграничение «удалено автором» vs «удалено администратором» в UI
- Временно́е окно для редактирования

## Technical Constraints

### Необходимые изменения схемы

Таблица `comments` требует:
1. `deleted_at DATETIME` — nullable; устанавливается при удалении, никогда не сбрасывается
2. FK `author_id`: **убрать `ON DELETE CASCADE`** — удаление аккаунта должно мягко удалять комментарии через сервисный слой, а не каскадным физическим удалением

### Хранение данных

- Текст и `author_id` сохраняются в базе данных после мягкого удаления
- Только API-слой скрывает их от ответов клиентам
- Это позволяет в будущем реализовать журнал аудита

### Безопасность

- Эндпоинты удаления и редактирования должны проверять личность вызывающего (Firebase Auth)
- Авторы могут изменять только собственные комментарии
- Администраторы обходят проверку владения

## MVP Scope

**В MVP:**
- Эндпоинт мягкого удаления (`DELETE /comments/{id}`)
- Эндпоинт редактирования (`PATCH /comments/{id}`)
- API не возвращает поля `text` и `author`, когда `deleted_at IS NOT NULL`
- Удаление аккаунта каскадно мягко удаляет комментарии (через сервисный слой)

**После MVP:**
- Инструменты модерации
- Журнал аудита администратора (кто удалил и когда)

## Risk Assessment

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Накопление осиротевших нод-плейсхолдеров со временем | Низкая | Низкое | Допустимо — целостность дерева важнее |
| Удаление аккаунта не охватывает soft delete комментариев | Средняя | Высокое | Обернуть в транзакцию на сервисном слое |
| Клиент рендерит текст удалённого комментария (утечка через API) | Низкая | Высокое | Фильтрация полей на уровне сериализации, а не в каждом обработчике |
