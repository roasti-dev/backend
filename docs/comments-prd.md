## Executive Summary

Комментарии — это кросс-ресурсная фича: пользователи могут обсуждать как посты, так и рецепты. Оба типа ресурсов используют единую систему комментариев с одинаковой моделью данных, правами доступа и поведением. Конкретные решения по threading-модели и жизненному циклу (удаление/редактирование) задокументированы в отдельных ADR-документах и здесь не повторяются.

---

## Problem Statement

**Текущая ситуация:** Комментарии впервые появились как часть постов и описаны в `posts-prd.md`. Аналогичная функциональность реализована для рецептов, но нигде явно не описана. Единых правил нет.

**Предлагаемое решение:** Выделить комментарии в самостоятельную фичу с отдельным PRD, описывающим общие правила для всех ресурсов.

## Supported Resource Types

Комментарии поддерживаются для следующих типов ресурсов:

| Тип ресурса | `target_type` |
|-------------|---------------|
| Пост | `post` |
| Рецепт | `recipe` |

Добавление нового ресурса не требует изменений в схеме комментариев — достаточно добавить новый `target_type`.

## User Stories & Acceptance Criteria

### Story 1: Добавить комментарий

**As a** авторизованный пользователь  
**I want to** оставить комментарий к посту или рецепту  
**So that** участвовать в обсуждении

**Acceptance Criteria:**
- [ ] Любой авторизованный пользователь может комментировать, включая автора ресурса
- [ ] Комментарий к приватному рецепту может оставить только его автор
- [ ] Успешное создание возвращает объект комментария (201)
- [ ] Комментарий может быть ответом на другой комментарий (`parent_id`)

### Story 2: Читать комментарии

**As a** пользователь  
**I want to** видеть комментарии к посту или рецепту  
**So that** следить за обсуждением

**Acceptance Criteria:**
- [ ] Комментарии возвращаются пагинированным списком тредов (`CommentThread`)
- [ ] Каждый тред — рутовый комментарий с плоским списком replies
- [ ] Replies включают все уровни вложенности (recursive), отсортированные по `created_at ASC`
- [ ] `parent_id` сохраняется в ответе для рендеринга @mention на клиенте

### Story 3: Редактировать комментарий

**As a** автор комментария  
**I want to** исправить свой комментарий  
**So that** скорректировать сказанное

**Acceptance Criteria:**
- [ ] Только автор может редактировать свой комментарий (403 для остальных)
- [ ] Нет временного ограничения на редактирование
- [ ] Удалённый комментарий нельзя редактировать
- [ ] `updated_at` обновляется при каждом редактировании

### Story 4: Удалить комментарий

**As a** автор комментария  
**I want to** удалить свой комментарий  
**So that** он больше не был виден другим

**Acceptance Criteria:**
- [ ] Только автор может удалить свой комментарий (403 для остальных)
- [ ] Удаление мягкое (soft delete): комментарий остаётся в дереве как tombstone
- [ ] Tombstone: `text` и `author` скрыты, `is_deleted: true`; `id`, `parent_id`, `created_at` остаются
- [ ] Replies под удалённым комментарием сохраняются видимыми
- [ ] Удаление необратимо — restore-эндпоинта нет

## Functional Requirements

### Права доступа

| Действие | Кто может |
|----------|-----------|
| Создать комментарий | Любой авторизованный пользователь (автор приватного ресурса — только он сам) |
| Читать комментарии | Любой авторизованный пользователь |
| Редактировать комментарий | Только автор комментария |
| Удалить комментарий | Только автор комментария |

### Threading-модель

Плоская модель в стиле Instagram: все replies под рутовым комментарием возвращаются единым плоским списком с `parent_id`. Подробности в `comment-threading-prd.md`.

### Жизненный цикл удаления

Soft-delete с tombstone-нодой. Подробности в `comment-lifecycle-prd.md`.

## API Contract

| Метод | Путь | Статус | Описание |
|-------|------|--------|----------|
| `POST` | `/api/v1/posts/{post_id}/comments` | 201 | Добавить комментарий к посту |
| `GET` | `/api/v1/posts/{post_id}/comments` | 200 | Комментарии поста (пагинация) |
| `POST` | `/api/v1/recipes/{recipe_id}/comments` | 201 | Добавить комментарий к рецепту |
| `GET` | `/api/v1/recipes/{recipe_id}/comments` | 200 | Комментарии рецепта (пагинация) |
| `PATCH` | `/api/v1/comments/{comment_id}` | 200 | Редактировать комментарий |
| `DELETE` | `/api/v1/comments/{comment_id}` | 204 | Удалить комментарий |

Эндпоинты редактирования и удаления общие — не привязаны к типу ресурса.

---

## Out of Scope (v1)

- Удаление комментария автором ресурса (только автор комментария в v1)
- Лайки на комментарии
- Упоминания (@username) на стороне бэкенда — только `parent_id` для клиента
- Уведомления при ответе на конкретный reply (только уведомление автору ресурса)
- Пагинация replies внутри треда
- Ограничение глубины вложенности

## Future Considerations

- Удаление комментариев автором ресурса
- Лайки на комментарии

- Уведомления при ответе на reply (не только автору ресурса)
- Расширенная модерация (репорты, админское удаление)
