## Executive Summary

Рецепты в Roasti сейчас не содержат структурированной информации о зерне — сорте, происхождении, обжарщике. Пользователи не могут делиться этой информацией в стандартизированном виде, что обедняет контент и снижает ценность рецептов как источника знаний о кофе.

Фича добавляет глобальный каталог зёрен: любой пользователь может добавить зерно с описанием происхождения, типом обжарки и дескрипторами, а затем привязать его к рецепту. Каталог разделяемый — одно зерно могут использовать несколько авторов в своих рецептах.

---

## Problem Statement

**Текущая ситуация**: Информация о зерне либо отсутствует в рецепте, либо хранится в свободном тексте (в описании). Нет стандартизированного способа указать происхождение, обжарщика или вкусовой профиль.

**Предлагаемое решение**: Глобальный каталог зёрен с CRUD операциями и поиском, с возможностью привязать зерно к рецепту при его создании или редактировании.

**Ожидаемый эффект**: Рецепты становятся более информативными; пользователи могут находить рецепты для конкретного зерна или обжарщика.

---

## Success Metrics

Метрики запланированы к определению после запуска. Функциональный успех MVP:

- Зерно создаётся, редактируется и мягко удаляется корректно
- Поиск возвращает релевантные результаты по названию и обжарщику
- Привязка к рецепту работает при создании и редактировании
- Удалённое зерно отображается как недоступное в существующих рецептах

## User Personas

### Primary: Автор рецепта

- **Роль**: Пользователь, публикующий рецепты на конкретных зёрнах
- **Цели**: Поделиться точной информацией о зерне; найти зерно из каталога или добавить новое
- **Pain Points**: Сейчас нет способа структурированно указать зерно — приходится писать в описании
- **Клиенты**: KMP (mobile), SvelteKit (web)

### Secondary: Читатель рецептов

- **Роль**: Пользователь, изучающий рецепты
- **Цели**: Узнать детали зерна, использованного в рецепте; найти рецепты для конкретного зерна
- **Pain Points**: Неполная или неструктурированная информация о зерне

## User Stories & Acceptance Criteria

### Story 1: Создание зерна

**As a** авторизованный пользователь  
**I want to** добавить зерно в каталог  
**So that** другие пользователи могут им воспользоваться в своих рецептах

**Acceptance Criteria:**
- [ ] Обязательные поля: название, тип обжарки (espresso / filter), обжарщик
- [ ] Опциональные поля: страна, регион, ферма/хозяйство, метод обработки, дескрипторы (список тегов)
- [ ] Зерно сразу появляется в глобальном каталоге
- [ ] Автор зерна зафиксирован при создании

### Story 2: Редактирование и удаление зерна

**As a** создатель зерна  
**I want to** редактировать или удалить своё зерно  
**So that** информация в каталоге оставалась актуальной

**Acceptance Criteria:**
- [ ] Только создатель может редактировать и удалять зерно
- [ ] Правки немедленно отражаются у всех, кто использует это зерно в рецептах
- [ ] Удаление — мягкое (soft delete): зерно исчезает из каталога
- [ ] Попытка редактировать/удалить чужое зерно возвращает 403
- [ ] В рецептах, где зерно было привязано, оно отображается с пометкой "недоступно" (аналогично recipe block со статусом unavailable)

### Story 3: Просмотр каталога и поиск

**As a** пользователь  
**I want to** найти нужное зерно в каталоге  
**So that** я мог привязать его к рецепту или просто узнать о нём

**Acceptance Criteria:**
- [ ] Поиск по названию и обжарщику (частичное совпадение)
- [ ] Мягко удалённые зёрна не появляются в результатах поиска
- [ ] Результаты пагинированы

### Story 4: Привязка зерна к рецепту

**As a** автор рецепта  
**I want to** указать зерно при создании или редактировании рецепта  
**So that** читатели знали какое зерно я использовал

**Acceptance Criteria:**
- [ ] Поле зерна в рецепте опциональное
- [ ] При заполнении: поиск по каталогу прямо в форме рецепта
- [ ] Если зерно не найдено — можно создать новое inline, не покидая форму рецепта
- [ ] Зерно можно отвязать от рецепта
- [ ] Один рецепт — одно зерно

### Story 5: Просмотр зерна

**As a** пользователь  
**I want to** открыть страницу зерна  
**So that** я мог увидеть все его детали

**Acceptance Criteria:**
- [ ] Доступен endpoint для получения зерна по ID
- [ ] Мягко удалённое зерно возвращает 404 при прямом запросе

## Functional Requirements

### Модель зерна

| Поле | Обязательность | Тип | Описание |
|---|---|---|---|
| `name` | Обязательное | string | Название зерна |
| `roast_type` | Обязательное | enum | `espresso` или `filter` |
| `roaster` | Обязательное | string | Название обжарщика (текст, задел на сущность в Phase 2) |
| `country` | Опциональное | string | Страна происхождения |
| `region` | Опциональное | string | Регион (Yirgacheffe, Huila и т.д.) |
| `farm` | Опциональное | string | Ферма или хозяйство |
| `process` | Опциональное | []string | Методы обработки (washed, natural, honey и т.д.) |
| `descriptors` | Опциональное | []string | Вкусовые дескрипторы (свободные теги) |
| `q_score` | Опциональное | float | Оценка Q-grading по шкале SCA (0–100) |
| `url` | Опциональное | string | Ссылка на страницу зерна у обжарщика |
| `image_id` | Опциональное | string | ID загруженного изображения (упаковка/зерно) |

### Права доступа

- Создавать зёрна может любой авторизованный пользователь
- Редактировать и удалять — только создатель
- Просматривать и использовать — любой авторизованный пользователь

### Soft Delete

- Удалённое зерно: `deleted_at` выставляется, из каталога исчезает
- В рецептах: зерно продолжает отображаться с флагом `is_available: false`
- Прямой запрос по ID удалённого зерна → 404

### Привязка к рецепту

- Поле `bean_id` в рецепте (опциональное, nullable)
- Устанавливается при создании или обновлении рецепта через `bean_id` в payload
- Inline-создание зерна из формы рецепта: отдельный POST на `/api/v1/beans`, результат сразу передаётся как `bean_id`

### API

| Метод | Путь | Описание |
|---|---|---|
| `POST` | `/api/v1/beans` | Создать зерно |
| `GET` | `/api/v1/beans` | Список/поиск зёрен (`?q=`, `?roaster=`) |
| `GET` | `/api/v1/beans/{bean_id}` | Получить зерно по ID |
| `PUT` | `/api/v1/beans/{bean_id}` | Обновить зерно (только создатель) |
| `DELETE` | `/api/v1/beans/{bean_id}` | Soft delete зерна (только создатель) |

Поле `bean` добавляется в модель `Recipe` (опциональный объект `BeanPreview`).

### Out of Scope (MVP)

- Сущность обжарщика с профилем, ссылкой, логотипом
- История изменений зерна
- Фильтрация по стране, методу обработки, дескрипторам
- Рейтинг / лайки зёрен
- Список рецептов, в которых используется данное зерно
- Публичная страница зерна (только данные через API)

## Technical Constraints

### Performance
- Поиск по `name` и `roaster` — полнотекстовый поиск через LIKE или SQLite FTS5 (на усмотрение при реализации)
- Результаты пагинированы (limit/offset, как везде в проекте)

### Security
- Все endpoints требуют аутентификации
- Редактирование/удаление чужого зерна → 403

### Integration

- **Recipes**: поле `bean_id` (nullable FK) добавляется в таблицу `recipes`, ответ рецепта включает `BeanPreview`
- **Soft delete**: рецепты отдают `bean` с `is_available: false`, если зерно удалено — аналогично существующему паттерну `recipe_status: unavailable` в post blocks
- **SQLite**: новая таблица `beans` с `deleted_at` для soft delete

### Technology Stack
- Go + SQLite + squirrel (как везде в проекте)
- OpenAPI-first: изменения в `api/spec.yaml` и `api/models.yaml`, затем `make oapi`

## MVP Scope & Phasing

### Phase 1: MVP

- Таблица `beans` с полной моделью
- CRUD endpoints для зёрен
- Поиск по названию и обжарщику
- Привязка зерна к рецепту (поле `bean_id` в recipes)
- Отображение удалённого зерна как недоступного в рецептах

### Phase 2: Обжарщик как сущность

- Отдельная таблица `roasters` (название, страна, ссылка, логотип)
- Зерно ссылается на `roaster_id` вместо текстового поля
- Миграция существующих текстовых значений

### Future Considerations

- Комплексный поиск (страна, метод обработки, дескрипторы)
- Рецепты по зерну
- Лайки / закладки зёрен
- Публичные страницы зёрен в UI

## Risk Assessment

| Риск | Вероятность | Влияние | Митигация |
|---|---|---|---|
| Дубли зёрен в каталоге | Средняя | Низкое | Поиск перед созданием снижает риск; дедупликация — в будущем |
| Поломка рецептов при удалении зерна | Низкая | Высокое | Soft delete + is_available флаг, FK без каскадного удаления |
| Миграция roaster с текста на сущность | Средняя | Средняя | Поле `roaster` (строка) изолировано — замена на `roaster_id` не затронет остальную модель |

## Dependencies & Blockers

**Dependencies:**
- Таблица `recipes` расширяется полем `bean_id` (nullable)
- Модель `Recipe` в API получает опциональное поле `bean` типа `BeanPreview`
- Паттерн soft delete с `is_available` флагом уже есть в проекте (post blocks)

**Known Blockers:** нет

## Appendix

### Glossary

- **Зерно (Bean)** — запись в каталоге с описанием конкретного кофейного зерна
- **Обжарщик (Roaster)** — компания или человек, обжаривший зерно
- **Soft delete** — логическое удаление через выставление `deleted_at`, без физического удаления из БД
- **Дескриптор** — вкусовой тег (например: "клубника", "шоколад", "цитрус")
- **BeanPreview** — краткое представление зерна, встраиваемое в ответ рецепта
